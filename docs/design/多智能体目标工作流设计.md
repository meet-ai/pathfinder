# 多智能体目标工作流设计

> 基于「发布任务 → 规划探索 → 并行/串行执行 → 按需生成 Skill/Tool → 进度追踪与中止 → 总结沉淀」的设想，整理为**目标工作流**，并映射到现有框架能力与实现思路。  
> 文档时间：2026-03

**语言：** [English](workflow-design.md) · [中文](多智能体目标工作流设计.md)

---

## 一、目标工作流（你的想法）

| 阶段 | 内容 |
|------|------|
| **1. 发布任务** | 用户/系统提交一个高层目标（如「完成某需求文档并给出实现方案」）。 |
| **2. 任务规划与探索** | 对目标做分解、依赖分析、探索可行路径；产出任务 DAG 或执行计划（谁做、顺序/并行）。 |
| **3. 执行与派发** | 按计划**并行或串行**执行子任务，把每项任务交给**合适的 agent**（如前端给 frontend-developer、安全给 security-engineer）。 |
| **4a. Skill 缺失时** | 若执行中发现所需 **Skill 不存在**，由（某）agent 考虑**生成 Skill**，并交给对应 agent 使用。 |
| **4b. Tool 缺失时** | 若所需 **Tool 不存在**，由（某）agent 考虑**生成 Tool**，并交给对应 agent 使用。 |
| **4c. 进度与管控** | **进度追踪**：随时反馈进度；**响应与反馈**；**中止不合理任务**（超时、无效、风险高时停止）。 |
| **5. 整理与总结** | 任务结束后**整理产出**、**总结经验**（可写文档、更新知识库、记录可复用 Skill/Tool）。 |

## 二、与现有框架的能力映射

### 2.1 阶段 1–3：发布任务、规划探索、执行派发

| 阶段 | 框架能力 | 说明 |
|------|----------|------|
| **1. 发布任务** | 各框架均支持 | 入口 = 用户消息 / API 调用（LangGraph invoke、CrewAI kickoff、agency-swarm get_response、OpenClaw agent --message）。 |
| **2. 任务规划与探索** | **LangGraph**、**CrewAI**、**AutoGen/AG2** | **LangGraph**：专长；用「规划节点」+ 条件边做分解与探索，state 里存 plan、子任务列表、依赖。**CrewAI**：Task 即子任务，Process 定顺序/层级，可加「规划 Agent」先产出 Task 列表。**AutoGen/AG2**：Manager 或专门 Planner agent 通过多轮对话做规划。 |
| **3. 并行/串行执行、派发 agent** | **LangGraph**、**CrewAI**、**agency-swarm**、**OpenClaw** | **LangGraph**：节点 = 执行步骤，不同节点调不同 OpenClaw agent 或 acpx；图可做并行分支。**CrewAI**：Task.agent 指定谁做，Process.sequential/hierarchical 定顺序。**agency-swarm**：communication_flows 定谁可被派活。**OpenClaw**：bindings 按渠道派；若上层用 LangGraph/CrewAI，则「派发」= 调 `openclaw:<agent-id>` 或 openclaw-sdk。 |

### 2.2 阶段 4a / 4b：Skill / Tool 缺失时生成并交付

| 能力 | 现状与实现思路 |
|------|----------------|
| **Skill 不存在 → 生成并交给 agent** | **OpenClaw**：Skill 有固定格式（如 SKILL.md、目录结构），可由「编排层」或「专用 agent」生成 Skill 包，再通过 ClawHub 安装或拷贝到 agent workspace/skills 目录；需一个**能写文件、能调安装流程**的节点或工具。**Cursor**：Subagent 的「能力」在 prompt/agents 目录，生成 = 生成新 .md 或规则并落盘。实现：LangGraph 的「技能生成节点」或 CrewAI 的「技能生成 Task」里调 LLM + 写文件 + 调用 `npx clawhub install` 或等价 API。 |
| **Tool 不存在 → 生成并交给 agent** | **OpenClaw / agency-swarm / CrewAI**：Tool 多为代码（函数/OpenAPI schema）。实现：专用「工具生成」节点/Agent 产出代码或 schema，写入项目或注册到框架（如 agency-swarm 的 BaseTool、CrewAI 的 tools 列表）；再将该 tool 绑定到目标 agent（下一轮执行时生效）。**Cursor/编码 agent**：tool ≈ 可执行能力；生成 = 生成脚本或 MCP 配置，由 acpx/OpenClaw 侧加载。 |

**要点**：4a/4b 在多数框架中**没有现成「一键生成并下发」**，需要你设计一个**显式步骤**（例如「检查 Skill/Tool 是否存在 → 若否则进入生成子图 → 写入/安装 → 再派给对应 agent」），用 LangGraph 子图或 CrewAI 的层级 Process 实现最自然。

### 2.3 阶段 4c：进度追踪、反馈、中止不合理任务

| 能力 | 现状与实现思路 |
|------|----------------|
| **进度追踪** | **LangGraph**：state 里维护「已完成/进行中/待办」子任务列表；节点可更新 state；支持 checkpoint，可持久化进度。**CrewAI**：Task 有状态；callbacks/事件可上报进度。**OpenClaw**：会话与执行在 Gateway 侧，可结合 Dashboard 或自建 API 查状态。实现：在编排层维护 `task_id → status, agent, started_at, result`，节点开始/结束或超时回调时更新。 |
| **随时响应和反馈** | 流式输出（LangGraph stream、CrewAI 流式、OpenClaw 流式）+ 进度事件推送到前端或日志；可选 WebSocket/SSE 把「当前步骤、当前 agent、进度%」推给用户。 |
| **停止不合理任务** | **LangGraph**：可在条件边或「监督节点」里根据 state（超时、错误次数、用户取消标志）路由到「中止」节点，并终止后续节点执行（或标记跳过）。**CrewAI**：可设 timeout、guardrails，或在 callback 里抛错/置状态以中断。**OpenClaw**：会话可取消；编排层若用 LangGraph，可设「监督节点」定期检查并调用取消逻辑。实现：统一在 state 或外部存储维护「cancel_requested / timeout / max_retries」，每个执行节点前检查，若需中止则走「清理与总结」分支。 |

### 2.4 阶段 5：整理与总结经验

| 能力 | 实现思路 |
|------|----------|
| **整理产出** | 最后一节点/最后一 Task：汇总各子任务结果（从 state 或 Task 输出收集），生成报告、文档或结构化产出；可写回知识库或交给用户。 |
| **总结经验** | 可选「复盘节点」：分析成功/失败子任务，产出简短总结或建议；将**新生成的 Skill/Tool** 登记到目录或 ClawHub，便于下次复用（对应 4a/4b 的持久化）。 |

## 三、推荐整体架构（对齐你的 1–5）

在**不换你想法**的前提下，用现有组件尽量贴齐上述流程的推荐方式如下。

### 3.1 编排层：LangGraph 作为「总控」

- **入口**：用户消息或 API 触发，进入 LangGraph 的**规划节点**。
- **规划节点**：调用 LLM（或 OpenClaw 的 planner agent）做任务分解与探索，产出「子任务列表 + 依赖 + 建议执行的 agent」写入 state。
- **执行层**：  
  - **串行**：按依赖顺序，条件边依次进入「执行节点」。  
  - **并行**：对无依赖的一组子任务，用 LangGraph 的并行节点（或多分支）同时进入多个「执行节点」。  
- **执行节点**：根据 state 中的「建议 agent」调用 OpenClaw（`model: openclaw:<agent-id>` 或 openclaw-sdk）或 acpx（`acpx <agent> "任务"`），把子任务描述与上下文传入；执行结果写回 state。
- **Skill/Tool 缺失**：  
  - 执行节点或「检查节点」发现缺少 Skill/Tool 时，将 state 置为「需要生成 Skill/Tool」，路由到**生成节点**。  
  - **生成节点**：调用「生成 agent」（如 OpenClaw 上专用于写 Skill/Tool 的 agent），产出文件并执行安装/注册，再写回 state「已就绪」，重新路由回执行或新子任务。  
- **进度与中止**：  
  - state 中维护 `tasks_done`、`tasks_failed`、`cancel_requested`、`deadline`。  
  - 每步前后有**监督节点**（或条件边）：若超时或 `cancel_requested` 或失败次数过多，路由到**中止节点**（清理、写结论），再结束。  
  - 流式与事件：LangGraph 的 stream 或 checkpoint 回调里发出「当前步骤、进度、当前 agent」，供前端或日志做「随时反馈」。  
- **总结节点**：所有子任务结束或中止后，进入**整理节点**：汇总 state 中的结果，生成最终产出；可选「复盘」写总结与登记新 Skill/Tool。

这样，你的 1→2→3→4a/4b→4c→5 都在**一张图**里表达，便于扩展和调试。

### 3.2 Agent 层：OpenClaw + agency-agents / Cursor（acpx）

- **规划 / 执行 / 生成 Skill / 生成 Tool**：由 OpenClaw 上不同 agent 承担（如 `planner`、`frontend-developer`、`backend-architect`、`specialized-mcp-builder` 等）；LangGraph 节点通过 Gateway 的 `openclaw:<agent-id>` 或 openclaw-sdk 调用。
- **编码类任务**：需要 Cursor 时，可由 LangGraph 节点调 acpx（如 `acpx cursor exec "任务"`），或由 OpenClaw 某 agent 通过 Skill/工具调用 Cursor API。
- **Skill/Tool 的「交给对应 agent」**：生成后写入 OpenClaw workspace 的 skills 目录或 ClawHub，或写入项目 `.cursor/agents` 等，下一轮执行时该 agent 即可使用。

### 3.3 简化架构图（文字）

```text
[用户] 发布任务
    ↓
[LangGraph] 规划节点 → 探索与分解 → state: 子任务列表、依赖、建议 agent
    ↓
[LangGraph] 条件边：并行分支 | 串行顺序
    ↓
[执行节点] 按 state 选 agent → 调 OpenClaw(openclaw:<id>) 或 acpx
    ↓
  缺失 Skill? → [生成 Skill 节点] → 写 Skill、安装 → 再执行
  缺失 Tool?  → [生成 Tool 节点]  → 写 Tool、注册  → 再执行
  超时/取消?  → [监督] → [中止节点] → 清理、写结论
    ↓
[LangGraph] 汇总节点 → 整理产出、总结经验（可选登记 Skill/Tool）
    ↓
[用户] 得到结果与反馈
```

## 四、各阶段与框架对照小结

| 你的阶段 | 推荐实现方式 |
|----------|--------------|
| 1. 发布任务 | 任意框架入口（LangGraph invoke、CrewAI kickoff、OpenClaw agent --message 等）。 |
| 2. 任务规划与探索 | **LangGraph** 规划节点 + state；或 CrewAI 的「规划 Agent + 动态 Task」；或 AutoGen Manager + Planner agent。 |
| 3. 并行/串行执行、派发 agent | **LangGraph** 节点调 OpenClaw/acpx，图表达并行与串行；或 **CrewAI** Task.agent + Process。 |
| 4a. Skill 不存在 → 生成并交付 | **LangGraph** 生成节点（或 CrewAI 专用 Task）：调「生成 agent」写 SKILL.md 等，执行安装/拷贝，state 标记已就绪后再执行。 |
| 4b. Tool 不存在 → 生成并交付 | 同上，生成节点产出 Tool 代码/schema，注册到 OpenClaw/agency-swarm/CrewAI，再派给对应 agent。 |
| 4c. 进度追踪、反馈、中止 | **LangGraph** state + 监督节点/条件边 + 流式/checkpoint；CrewAI callbacks + timeout/guardrails。 |
| 5. 整理与总结经验 | 最后节点汇总 state、写报告；可选写知识库、登记 Skill/Tool 到 ClawHub 或目录。 |

## 五、仅用 OpenClaw 能否直接实现？

在**不引入 LangGraph/CrewAI 等外部编排**的前提下，只用 OpenClaw（Gateway + 多 Agent + Skill + 会话工具）能否实现你设想的 1–5 流程？结论如下。

### 5.1 能力逐项对照

| 阶段 | 能否直接用 OpenClaw 实现 | 说明 |
|------|---------------------------|------|
| **1. 发布任务** | ✅ **能** | 用户/系统通过 `openclaw agent --agent <id> --message "..."` 或入站消息（Slack/WebChat 等）提交；bindings 选 agent。 |
| **2. 任务规划与探索** | ⚠️ **靠一个「规划 Agent」** | 无内置 DAG/规划器。做法：设一个**规划 agent**（如 `planner`），在 AGENTS.md 里约定「先分解目标、产出子任务与建议执行的 agent 列表」；该 agent 用 **sessions_send** 或 **sessions_spawn** 把子任务派给其他 agent。规划质量依赖该 agent 的 prompt 与模型，非框架级保证。 |
| **3. 并行/串行执行、派发 agent** | ✅ **能** | **sessions_send**：当前 agent 发消息到指定 session（可指向另一 agent 的 main 或指定 session），可等待回复或 fire-and-forget。**sessions_spawn**：派发子任务到**子 agent**（可指定 `agentId`），异步执行、支持 `runTimeoutSeconds`，结果通过 announce 回传；多路 spawn = 并行。串行 = 顺序 sessions_send 或 spawn 后等结果再发下一项。社区 Skill **dispatch-multiple-agents**、**Clawflow** 可复用（并行派发、DAG 依赖）。 |
| **4a. Skill 不存在 → 生成并交付** | ⚠️ **需自建流程与工具** | OpenClaw 无「一键生成 Skill」内置。做法：规划 agent 或专用「生成 agent」在发现缺 Skill 时，用**写文件 + 执行 shell** 的工具（或现有 Skill）生成 SKILL.md 等，再执行 `npx clawhub install <path>` 或拷贝到目标 agent 的 workspace/skills；需在 AGENTS.md 里约定「何时生成、如何安装、如何通知目标 agent」。 |
| **4b. Tool 不存在 → 生成并交付** | ⚠️ **需自建流程与工具** | 同上，由某 agent 生成 Tool 代码或 schema，写入 workspace 或通过 Skill 注册；OpenClaw 的 tool 多来自 Skill/配置，**动态注册**可能需自写 Skill 或插件，或把新 tool 写进某 agent 的 skills 目录后重载。 |
| **4c. 进度追踪、反馈、中止** | ⚠️ **部分能** | **进度**：**sessions_list** / **sessions_history** 可查会话与最近消息；无内置「任务级进度表」，需规划 agent 或某 agent 自己维护（如写 MEMORY.md、或调外部 API 存 task_id/status）。**反馈**：流式回复、deliver 到 channel 即「反馈」；若要做成「当前步骤/进度%」，需 agent 在回复或工具里显式输出。**中止**：**sessions_spawn** 的 `runTimeoutSeconds` 可超时终止子任务；**无**官方「用户/监督者主动取消某次 run」的会话级 API（需查 Gateway 是否支持 cancel）。子 agent 仅单层（子 agent 不能再 spawn），复杂 DAG 需多轮 send/spawn 由主 agent 协调。 |
| **5. 整理与总结经验** | ✅ **能** | 规划 agent（或最后被派到的 agent）在流程末尾做汇总、写 MEMORY.md 或总结文件；可选把新 Skill 登记到 ClawHub 或团队目录，供下次复用。 |

### 5.2 结论与建议

- **能直接用 OpenClaw 实现的部分**：1 发布任务、3 派发与并行/串行执行（sessions_send / sessions_spawn + 多 agent）、5 整理与总结。  
- **需要「在 OpenClaw 内自建」的部分**：2 任务规划（依赖一个规划 agent 的 prompt 与 sessions_send/spawn）、4a/4b 生成 Skill/Tool（依赖具备写文件/执行命令的 agent 与约定流程）、4c 进度与中止（依赖自维护状态与 runTimeoutSeconds，主动取消需看 Gateway 能力）。  
- **若希望「任务 DAG、进度表、统一中止、生成节点」等都有现成抽象**：仍建议用 **LangGraph（或 CrewAI）做编排层**，用 OpenClaw 做执行层（每个节点调 openclaw:<agent-id> 或 sessions_send/spawn）；这样 2/4a/4b/4c 由编排层保证，OpenClaw 专注「多 agent 执行与 Skill/Tool 承载」。  
- **若坚持只用 OpenClaw**：可行路径是：**一个规划 agent + sessions_send/sessions_spawn + dispatch-multiple-agents/Clawflow 类 Skill + 自约定「生成 Skill/Tool」的流程与权限**；进度与中止用 runTimeoutSeconds 与自维护状态（如 MEMORY 或外部存储）补足，接受「无原生 DAG、无原生任务级取消」的限制。

### 5.3 相关 OpenClaw 能力速查

- **派发到其他 agent**：[Session Tools](https://docs.openclaw.ai/concepts/session-tool)（sessions_send、sessions_spawn）；spawn 支持 `agentId`、`runTimeoutSeconds`。
- **多 agent 并行**：社区 Skill **dispatch-multiple-agents**、**Clawflow**（DAG）；主仓多 workspace + bindings。
- **Skill/Tool**：workspace 内 `skills/`、ClawHub 安装、AGENTS.md 约定行为。
