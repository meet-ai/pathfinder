# 多智能体目标工作流 — 特性列表

> 基于《多智能体目标工作流设计》整理，便于产品/开发跟踪与验收。  
> 每条特性可交付、可验收；按工作流阶段与能力域分组。

---

## 使用流程（其一）：直接发布 + TUI

**约定**：用户通过 CLI 一次发布任务，本地启动 TUI 展示该任务的执行过程与结果。

| 步骤 | 用户操作 | 系统行为 |
|------|----------|----------|
| 1 | 执行 `pathfinder -m "目标描述"`（或等价 `run --message "..."`） | 提交目标到编排层，获得 run_id/会话；进程不退出。 |
| 2 | — | 启动 **TUI**，绑定该 run_id，订阅进度与流式输出。 |
| 3 | 在 TUI 中查看 / 必要时取消 | 展示：当前阶段（规划 / 执行 / 总结）、当前步骤与 agent、子任务进度（如 3/7）、流式日志；支持取消。 |
| 4 | 任务结束或取消后查看结果 | TUI 展示最终汇总或错误信息；退出 TUI 后进程结束。 |

**验收**：一条命令发任务后，本地自动弹出 TUI，能实时看到执行阶段与进度，并可在 TUI 中取消或查看最终结果。

## 一、任务发布与入口

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F1.1 | 支持通过**统一 API 入口**提交高层目标并进入工作流 | 提供 HTTP/API 调用（如 LangGraph invoke、OpenClaw agent --message），传入目标描述即触发工作流；验收：调用成功并返回 run_id 或流式句柄。 |
| F1.2 | 支持通过**消息渠道**提交任务（如 Chat、Slack、WebChat） | 通过 bindings 将用户消息路由到指定 agent 或编排入口；验收：在配置的渠道发送消息可触发任务。 |
| F1.3 | 任务提交时支持**可选参数**（如超时、优先级、指定 agent 池） | 入口接受扩展参数，并写入 state 或配置供后续节点使用；验收：参数在规划/执行/中止逻辑中生效。 |

## 二、规划与探索

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F2.1 | 对用户目标做**任务分解**，产出结构化子任务列表 | 规划节点（或规划 Agent）调用 LLM/planner 将目标拆为多条子任务；验收：给定一项目标，产出子任务列表（含标题/描述）。 |
| F2.2 | 对子任务做**依赖分析**，产出 DAG 或执行顺序 | 明确子任务间依赖（如 A→B→C 或 A、B 并行再 C）；验收：输出可被执行层消费的依赖关系（如 edges 或 ordered list）。 |
| F2.3 | 为每项子任务**分派建议执行的 agent** | 规划结果中每项子任务对应建议 agent（如 frontend-developer、security-engineer）；验收：执行节点能根据该字段选择 openclaw:<agent-id> 或 acpx 目标。 |
| F2.4 | 规划结果**写入编排层 state**，供执行与监督节点使用 | 子任务列表、依赖、建议 agent 写入 LangGraph state 或等价结构；验收：执行节点与监督节点能读取并据此调度。 |

## 三、执行与派发

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F3.1 | 按依赖顺序**串行执行**子任务 | 有依赖的子任务按拓扑序依次进入执行节点；验收：顺序符合 DAG，且前一任务结果可传入下一任务。 |
| F3.2 | 对无依赖子任务**并行执行** | 同一「层」的子任务通过并行节点或多分支同时执行；验收：并发执行且结果正确汇总到 state。 |
| F3.3 | 执行节点根据 state **派发到对应 agent** | 通过 openclaw:<agent-id>、openclaw-sdk 或 acpx 将子任务描述与上下文传给指定 agent；验收：任务实际由预期 agent 执行并返回结果。 |
| F3.4 | 将**执行结果写回 state**，供汇总与条件边使用 | 每个执行节点完成后把结果（含成功/失败、输出内容）写入 state；验收：总结节点与条件边能读取到完整结果。 |

## 四、Skill / Tool 按需生成

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F4.1 | **检测「所需 Skill 不存在」**并路由到生成流程 | 执行节点或检查节点在发现缺 Skill 时，将 state 置为「需生成 Skill」并路由到生成节点；验收：缺 Skill 时不会卡死，会进入生成分支。 |
| F4.2 | **生成符合规范的 Skill 包**（如 SKILL.md、目录结构） | 生成节点（或专用 agent）产出 Skill 文件并落盘；验收：产出物符合 OpenClaw/Cursor 等目标平台的 Skill 格式。 |
| F4.3 | **安装或部署 Skill** 到目标 agent（如 clawhub install / 拷贝到 skills） | 生成后执行安装或拷贝到 workspace/skills，使目标 agent 可见；验收：目标 agent 在下一轮执行中能使用该 Skill。 |
| F4.4 | **检测「所需 Tool 不存在」**并路由到生成流程 | 与 F4.1 对称，缺 Tool 时路由到 Tool 生成节点；验收：缺 Tool 时进入生成分支。 |
| F4.5 | **生成 Tool 并注册**到框架并绑定到目标 agent | 生成节点产出 Tool 代码或 OpenAPI schema，注册到 OpenClaw/agency-swarm/CrewAI 等，并绑定到对应 agent；验收：下一轮执行时该 agent 可调用新 Tool。 |
| F4.6 | 生成完成后**重新调度执行**（重试原任务或新子任务） | Skill/Tool 就绪后 state 更新，重新路由回执行节点；验收：原失败任务可重试或新任务能使用新能力。 |

## 五、进度与中止

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F5.1 | **维护任务级进度**（已完成/进行中/待办）并支持持久化 | state 或外部存储维护 task_id → status, agent, started_at, result；支持 checkpoint 等持久化；验收：中断后恢复能正确反映进度。 |
| F5.2 | **流式输出与进度事件**（当前步骤、当前 agent、进度%） | 通过 LangGraph stream、SSE 或 WebSocket 推送「当前步骤、当前 agent、进度%」到前端或日志；验收：客户端能实时看到进度变化。 |
| F5.3 | **超时检测与自动中止** | 单任务或整体 run 支持 deadline/timeout，超时后路由到中止节点；验收：超时后不再执行新子任务并进入清理与总结。 |
| F5.4 | **用户/监督者主动取消**任务并走清理分支 | 支持 cancel_requested 或等价标志，监督节点或条件边检测后路由到中止节点；验收：用户取消后流程终止并产出可用的总结。 |
| F5.5 | **失败/风险策略**：失败次数过多或风险过高时中止 | 在 state 或监督节点中根据失败次数、错误类型等决定是否中止；验收：满足条件时进入中止与总结，不无限重试。 |

## 六、总结与沉淀

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F6.1 | **汇总各子任务结果**，生成最终报告或结构化产出 | 总结节点从 state 收集所有子任务结果，生成报告、文档或 JSON 等；验收：用户或下游系统能拿到完整汇总产出。 |
| F6.2 | 支持将**最终产出写回知识库**或交付指定端 | 可选：把报告写入知识库、上传到指定存储或通过 API 回调交付；验收：配置后产出能写入目标位置。 |
| F6.3 | **可选复盘节点**：分析成功/失败子任务，产出简短总结或建议 | 可选：在总结前增加复盘逻辑，输出「经验总结」或「改进建议」；验收：有开关时能产出复盘内容。 |
| F6.4 | **将新生成的 Skill/Tool 登记**到目录或 ClawHub 便于复用 | 4a/4b 中生成的 Skill/Tool 在流程结束时可选登记到团队目录或 ClawHub；验收：登记后在其他 run 或 agent 中可发现并复用。 |

## 七、架构与集成

| ID | 特性描述 | 说明 / 验收标准 |
|----|----------|-----------------|
| F7.1 | **编排层与执行层分离**：编排（LangGraph/CrewAI）调执行（OpenClaw/acpx） | 架构上编排图负责规划、分支、监督、总结，执行节点仅负责调 agent；验收：文档与示例体现分层，且可替换执行层实现。 |
| F7.2 | **关键节点可配置/可扩展**（规划、执行、生成、监督、总结） | 规划节点、执行节点、Skill/Tool 生成节点、监督节点、总结节点可通过配置或扩展点替换；验收：至少一种方式可替换或扩展各节点逻辑。 |
| F7.3 | **与 OpenClaw Gateway、ClawHub、Cursor acpx 的集成说明与示例** | 提供如何用 openclaw:<agent-id>、clawhub、acpx 与编排层对接的文档与可运行示例；验收：按文档能完成端到端部署并跑通。 |
| F7.4 | **「仅 OpenClaw」模式**：在无 LangGraph 时用规划 Agent + sessions_send/spawn 实现子集 | 文档与示例说明仅用 OpenClaw（规划 agent + dispatch-multiple-agents/Clawflow 类 Skill）可实现哪些特性及限制；验收：能列出能力对照与取舍说明。 |

## 八、支撑模块（按 DDD 职责分模块）

按**限界上下文**划分支撑能力，每个模块单一职责、边界清晰；上层特性依赖这些模块的端口接口。

### 8.1 能力目录（Capability Catalog）

**职责**：回答「谁可用、会什么」；执行体与能力的发现、检索、登记。

| 子模块 | 能力简述 | 依赖的特性 |
|--------|----------|------------|
| **Agent 发现** | 可用 agent 列表与元数据（id、能力描述、标签/角色）；按 agent 池或能力过滤；对接 Gateway/ClawHub/本地配置。 | F1.3、F2.3、F3.3、F4.3/F4.5、F7.3/F7.4 |
| **Skill/Tool 目录与存在性** | 查询某 agent 是否具备某 Skill/Tool；目录列表与检索；登记/安装（ClawHub install、落盘 skills）。 | F4.1、F4.2/F4.3、F4.4/F4.5、F6.4 |

**端口**：ListAgents(filter)、GetAgent(id)、HasSkill(agent, skill)、ListSkills(agent)、RegisterSkill/InstallSkill、Tool 对称接口。

---

### 8.2 规划契约（Planning Contract）

**职责**：规划产出的结构定义、校验及与编排层 state 的读写约定；规划节点产出，执行/监督/总结节点消费。

| 子模块 | 能力简述 | 依赖的特性 |
|--------|----------|------------|
| **规划结果契约** | 子任务列表、依赖（DAG/顺序）、建议 agent 等结构定义与校验；state 读写契约。 | F2.1、F2.2、F2.3、F2.4 |

**端口**：PlanSchema（子任务、依赖、建议 agent）、Validate(plan)、State 读写约定（与编排层一致）。

---

### 8.3 运行时（Runtime）

**职责**：run 生命周期、派发到执行体、流式输出、取消。

| 子模块 | 能力简述 | 依赖的特性 |
|--------|----------|------------|
| **运行时会话与派发** | 创建 run、返回 run_id/流式句柄；派发子任务到指定 agent（openclaw:\<id\>/acpx）；流式/SSE；取消 run。 | F1.1、F3.3、F5.2、F5.4 |

**端口**：CreateRun(params)、Dispatch(run_id, task, agent_id)、Stream(run_id)、Cancel(run_id)。

---

### 8.4 执行状态（Execution State）

**职责**：任务级进度、checkpoint、持久化与恢复；供监督节点与总结节点读取。

| 子模块 | 能力简述 | 依赖的特性 |
|--------|----------|------------|
| **进度与状态持久化** | task_id → status, agent, started_at, result；checkpoint/持久化与恢复。 | F2.4、F3.4、F5.1、F6.1 |

**端口**：WriteTaskProgress(run_id, task_id, status, result)、ReadProgress(run_id)、Checkpoint(run_id)、Restore(run_id)。

---

## 使用说明

- **ID**：用于 Issue、Project 或需求管理系统中的引用（如 `F3.2`）。
- **验收标准**：可作为 DoD（Definition of Done）或测试用例的输入。
- **优先级与迭代**：可按阶段分批实现（如先 1→2→3→5，再 4a/4b、4c 增强）。
